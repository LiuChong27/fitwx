# 小程序管理员操作手册

本文档基于当前项目代码整理，目标是让第一次做小程序的人也能按步骤完成管理员开通、内容审核、举报处理、发布上线和日常运维。

适用项目：

- 前端：`uni-app`
- 云端：`uniCloud`
- 管理员入口：`pages/admin/index`
- 管理员核心能力：
  - 动态审核
  - 举报处理
  - 聊天未读数修复（运维能力）

注意：本文只写“当前项目里已经实现”的管理员能力，不额外虚构不存在的后台功能。

## 1. 管理员现在能做什么

根据当前项目代码，管理员已经具备以下能力。

### 1.1 已实现的前台管理员功能

1. 进入管理员面板
2. 查看待审核动态列表
3. 审核通过动态
4. 驳回动态，并填写驳回原因
5. 查看举报列表
6. 按状态筛选举报
7. 查看举报详情
8. 对举报执行处置动作
9. 下架被举报动态
10. 警告被举报用户
11. 驳回举报
12. 处理完成后，给相关用户发送系统通知

### 1.2 已实现但不在前台页面里的管理员/运维能力

1. 重建聊天会话未读数
2. 发布和更新云函数
3. 发布数据库 schema 和索引
4. 为账号开通 `admin` 角色
5. 调整内容安全规则表 `fit-content-security-rules`

说明：

- `fit-content-security-rules` 目前是数据库级配置，不是小程序里的可视化页面。
- 如果你后面想做真正的后台管理系统，可以在后续迭代中再加“管理员用户管理”“内容安全规则管理”“运营数据看板”等页面。

## 3. 先理解管理员相关状态

这一步很重要。你只要把几个状态记住，后面操作会很顺。

### 3.1 动态 `fit-feeds.status`

- `0`：待审核
- `1`：正常展示
- `2`：已删除 / 已下架

### 3.2 动态内容安全状态 `fit-feeds.moderation_status`

- `pass`：通过
- `review`：进入人工审核
- `block`：拦截

### 3.3 举报 `fit-reports.status`

- `0`：待处理
- `1`：已处理
- `2`：已驳回

### 3.4 举报处置动作 `fit-reports.handle_action`

- `pending`：待处理
- `remove_feed`：下架动态
- `warn_user`：警告用户
- `dismiss`：驳回举报

## 4. 管理员完整工作流总览

当前项目的管理员工作流，可以理解为下面这条链路。

### 4.1 动态审核链路

1. 用户发布动态
2. 系统先做内容安全检查
3. 如果结果是 `pass`，动态直接进入公开流
4. 如果结果是 `review`，动态进入“待审核”
5. 管理员在“动态审核”页处理
6. 审核通过后，动态公开展示
7. 审核驳回后，动态不公开，用户收到通知

### 4.2 举报处理链路

1. 用户举报动态或聊天对象
2. 系统记录举报单到 `fit-reports`
3. 管理员在“举报处理”页查看
4. 管理员打开详情
5. 选择“下架动态”/“警告用户”/“驳回举报”
6. 系统更新举报状态
7. 如涉及动态下架，则更新动态状态为 `2`
8. 系统给举报人、被处理用户发送通知

### 4.3 运维修复链路

1. 如果聊天未读数不准
2. 管理员或运维人员调用 `fit-maintenance-api`
3. 执行 `rebuildConversationUnreadMap`
4. 系统批量修复会话未读数字段

## 5. 第一次使用前，先做这些准备

如果你是第一次上线这个小程序，建议按下面顺序准备。

### 5.1 先准备账号

你至少需要两个账号：

1. 一个普通用户账号
2. 一个管理员账号

这样方便你自己完整测试“用户发布/举报 + 管理员审核/处理”这整套流程。

### 5.2 确认 `admin` 角色存在

项目里已经存在 `admin` 角色定义，位置是：

- `uniCloud-aliyun/database/uni-id-roles.init_data.json`

它的角色 ID 是：

- `admin`

### 5.3 给某个账号开通管理员权限

当前项目判断管理员身份的方式是：

- 当前登录用户的 `role` 或 `roles` 数组中包含 `admin`

也就是说，只要目标用户的角色列表里有 `admin`，设置页就会显示“管理员面板”，并且管理员云函数也允许调用。

#### 做法 A：用 uni-id 的管理员注册页

项目里已经包含：

- `uni_modules/uni-id-pages/pages/register/register-admin`

这个页面是 `uni-id-pages` 自带的管理员注册页。

适合场景：

- 你还没有管理员账号
- 你想先创建第一个超级管理员

注意：

- 这个页面是否直接开放给用户，需要你自己控制使用场景
- 一般只在开发/初始化阶段使用

#### 做法 B：直接在数据库里给用户加角色

这是最直接、最稳妥的方式。

操作步骤：

1. 打开 HBuilderX
2. 连接当前项目绑定的 uniCloud 服务空间
3. 进入数据库管理
4. 打开 `uni-id-users`
5. 找到你要设为管理员的用户记录
6. 找到用户的 `role` 字段
7. 把 `admin` 加入数组

示例：

```json
{
  "role": ["admin"]
}
```

如果原来已有其他角色，则保留原有角色，例如：

```json
{
  "role": ["user", "admin"]
}
```

修改完成后：

1. 重新登录这个账号
2. 进入“我的”
3. 打开“设置”
4. 确认已经出现“管理员面板”

## 6. 管理员每天实际要做什么

当前项目最主要的管理员日常工作只有两类：

1. 审核动态
2. 处理举报

下面我按“点哪里、看到什么、下一步做什么”的方式写。

## 7. 动态审核操作说明

### 7.1 什么时候需要审核动态

当用户发布的动态命中了内容安全规则中的“人工审核”条件时，动态会被保存为：

- `status = 0`
- `moderation_status = review`

这时候动态不会进入公开流，需要管理员处理。

### 7.2 管理员如何进入动态审核页

详细步骤：

1. 用管理员账号登录小程序
2. 进入底部导航“我的”
3. 点击“设置”
4. 找到“管理员面板”
5. 点击进入
6. 点击“动态审核”

如果你看不到“管理员面板”，通常说明：

1. 当前账号没有 `admin` 角色
2. 登录态没有刷新，重新登录一次

### 7.3 动态审核页能看到什么

动态审核页会显示：

1. 待审核数量
2. 刷新按钮
3. 待审核动态列表
4. 每条动态的封面、作者、运动类型、等级、发布时间、正文、位置
5. 两个操作按钮：
   - `通过`
   - `驳回`

### 7.4 审核通过怎么操作

详细步骤：

1. 打开“动态审核”页
2. 找到要处理的动态
3. 先看内容是否合规
4. 再看图片是否清晰、是否存在广告导流、低俗、违法违规等问题
5. 如果确认可以展示，点击“通过”
6. 等待页面提示“已通过”

系统会自动完成这些事：

1. 将 `fit-feeds.status` 更新为 `1`
2. 记录审核时间 `review_date`
3. 记录审核人 `review_by`
4. 给动态作者发送系统通知
5. 该动态之后可以出现在社区公开流中

### 7.5 驳回动态怎么操作

详细步骤：

1. 打开“动态审核”页
2. 找到要处理的动态
3. 点击“驳回”
4. 底部会弹出驳回输入框
5. 在输入框中填写驳回原因
6. 点击“确认驳回”
7. 等待页面提示“已驳回”

建议驳回原因写清楚，例如：

1. 内容不完整，请补充训练目的和具体描述
2. 图片不清晰，请重新上传
3. 包含广告导流信息，请修改后重新发布
4. 存在违规内容，无法通过审核

系统会自动完成这些事：

1. 将 `fit-feeds.status` 更新为 `2`
2. 写入 `review_remark`
3. 记录审核时间和审核人
4. 给动态作者发送“审核未通过”的通知

### 7.6 审核动态时的判断标准建议

你可以先按下面这套简单标准执行。

#### 可以通过

1. 内容真实、完整、可理解
2. 图片正常，没有明显违规
3. 没有广告导流
4. 没有辱骂、低俗、违法内容
5. 与健身/约练/训练相关

#### 建议驳回

1. 明显广告
2. 联系方式导流
3. 色情低俗
4. 违法违规
5. 空内容或无意义内容
6. 图片严重模糊或不适合展示

## 8. 举报处理操作说明

### 8.1 举报处理页解决什么问题

这个页面用于集中处理用户提交的举报单。

当前项目支持的举报对象：

1. 动态 `feed`
2. 聊天对象 `chat_user`

### 8.2 管理员如何进入举报处理页

详细步骤：

1. 用管理员账号登录小程序
2. 进入“我的”
3. 打开“设置”
4. 点击“管理员面板”
5. 点击“举报处理”

### 8.3 举报处理页能看到什么

页面上会显示：

1. 待处理数量
2. 已处理数量
3. 已驳回数量
4. 状态筛选：
   - 待处理
   - 已处理
   - 已驳回
   - 全部
5. 举报列表

每条举报卡片会显示：

1. 举报类型
2. 举报人
3. 举报时间
4. 当前状态
5. 举报原因
6. 被举报对象摘要
7. 被举报对象昵称

### 8.4 查看举报详情怎么操作

详细步骤：

1. 打开“举报处理”页
2. 在列表中点击任意一条举报
3. 底部弹出详情面板
4. 查看：
   - 举报对象
   - 举报内容摘要
   - 举报原因
   - 举报人
   - 举报补充说明
5. 如果该举报还没处理，你会看到备注输入框和操作按钮

### 8.5 举报有哪些处理动作

当前项目支持 3 种动作。

#### 动作 1：下架动态

适用场景：

1. 举报属实
2. 被举报对象是动态
3. 动态确实违规或不适合展示

操作步骤：

1. 打开举报详情
2. 在“处置备注”里填写说明
3. 点击“下架动态”
4. 等待系统提示“处理已完成”

系统会自动完成这些事：

1. 将这条动态的 `fit-feeds.status` 更新为 `2`
2. 将动态 `moderation_status` 标记为 `block`
3. 给举报人发送“举报已处理”通知
4. 给被处理用户发送“动态已下架”通知
5. 将和这个对象相关的其他待处理举报一并处理掉

#### 动作 2：警告用户

适用场景：

1. 举报有一定依据
2. 但暂时不需要直接下架动态
3. 或举报对象是聊天对象而非动态

操作步骤：

1. 打开举报详情
2. 填写处置备注
3. 点击“警告用户”
4. 等待系统提示处理完成

系统会自动完成这些事：

1. 举报状态变为已处理
2. 给举报人发送“平台已警告对方”的通知
3. 给被举报用户发送警告通知

#### 动作 3：驳回举报

适用场景：

1. 证据不足
2. 举报不成立
3. 举报属于误报

操作步骤：

1. 打开举报详情
2. 可选填写备注
3. 点击“驳回举报”
4. 等待处理成功提示

系统会自动完成这些事：

1. 举报状态变为 `2`
2. 处置动作标记为 `dismiss`
3. 给举报人发送“当前未发现足够违规证据”的通知

### 8.6 举报处理建议标准

#### 建议直接下架动态

1. 色情低俗
2. 赌博、诈骗、违法信息
3. 严重广告导流
4. 明显骚扰、辱骂、人身攻击

#### 建议先警告用户

1. 轻度不友好表达
2. 初次违规且情节较轻
3. 有争议但还不足以下架的内容

#### 建议驳回举报

1. 举报理由明显不成立
2. 无法找到被举报事实
3. 举报人恶意滥用举报功能

## 9. 普通用户发布后的内容为什么有时看不到

这个问题很多新项目都会遇到，你需要理解。

当前项目里，用户发动态时不是“一定立即公开”，而是先走内容安全检查。

### 9.1 系统会怎么处理

#### 情况 1：内容安全通过

系统会把动态写成：

- `status = 1`
- `moderation_status = pass`

这时用户发完就能在公开流看到。

#### 情况 2：需要人工审核

系统会把动态写成：

- `status = 0`
- `moderation_status = review`

这时动态会进入管理员的待审核列表。

#### 情况 3：内容直接违规

系统会拒绝发布，前端直接提示失败。

### 9.2 你应该怎么理解这个机制

简单说：

1. `pass`：自动过
2. `review`：管理员手动审
3. `block`：系统直接拦

## 10. 第一次完整演练建议

你第一次接手这个小程序，强烈建议按下面步骤完整跑一遍。

### 10.1 演练目标

确认这 4 件事都正常：

1. 管理员入口能进入
2. 动态审核能通过/驳回
3. 举报处理能完成
4. 通知能发送给用户

### 10.2 演练步骤

#### 第一步：准备两个账号

1. 普通账号 A
2. 管理员账号 B

#### 第二步：给账号 B 加 `admin` 角色

按本文第 5.3 节操作。

#### 第三步：普通账号 A 发布一条动态

1. 内容尽量正常
2. 再尝试一条容易触发审核的内容

#### 第四步：管理员账号 B 进入动态审核页

1. 检查是否看到待审核动态
2. 对其中一条执行“通过”
3. 对另一条执行“驳回”

#### 第五步：用账号 A 看通知

1. 进入通知中心
2. 查看是否收到审核结果通知

#### 第六步：账号 A 再提交一条举报

可以举报一条动态。

#### 第七步：管理员账号 B 进入举报处理页

1. 打开举报详情
2. 试一次“警告用户”
3. 再试一次“驳回举报”

#### 第八步：验证通知

1. 举报人是否收到处理结果通知
2. 被处理用户是否收到警告或下架通知

## 11. 小程序发布前，你要做哪些事

这里分成两部分理解：

1. 发布 uniCloud 后端
2. 发布微信小程序前端版本

这两个都要做。

## 12. 发布 uniCloud 的正确顺序

项目里已经有一份发布顺序文档：

- `docs/unicloud-release-order.md`

你第一次操作时，建议按下面更容易理解的顺序执行。

### 12.1 发布前准备

1. 先确认当前项目绑定的是正确的 uniCloud 服务空间
2. 先备份生产数据库
3. 确认 `manifest.json` 里的微信 `appid` 是正式项目的
4. 确认 schema 文件在 `uniCloud-aliyun/database`
5. 确认云函数在 `uniCloud-aliyun/cloudfunctions`

### 12.2 建议发布顺序

1. 发布数据库 schema
2. 发布数据库索引
3. 发布业务云函数
4. 发布管理员/运维云函数
5. 做冒烟测试
6. 如有需要，再执行未读数修复

### 12.3 这次项目重点要发布哪些

#### 数据库 schema

至少确认这些表已发布：

1. `fit-feeds`
2. `fit-reports`
3. `fit-notifications`
4. `fit-conversations`
5. `fit-messages`
6. `uni-id-users`
7. `uni-id-roles`
8. `uni-id-permissions`

#### 云函数

至少确认这些已发布：

1. `fit-discover-api`
2. `fit-report-api`
3. `fit-notification-api`
4. `fit-maintenance-api`
5. `fit-ucenter-api`
6. `fit-meet-api`

### 12.4 在 HBuilderX 里怎么发布云函数

详细步骤：

1. 打开 HBuilderX
2. 打开项目 `d:\fit`
3. 找到左侧 `uniCloud-aliyun`
4. 展开 `cloudfunctions`
5. 对每个目标云函数右键
6. 点击“上传并部署”
7. 等待部署完成
8. 查看控制台日志，确认没有启动报错

### 12.5 在 HBuilderX 里怎么发布数据库 schema

详细步骤：

1. 打开 `uniCloud-aliyun/database`
2. 找到对应 `.schema.json` 文件
3. 右键选择“上传 DB Schema”
4. 等待上传成功
5. 到 uniCloud 控制台或 HBuilderX 数据库面板确认字段和索引已经同步

### 12.6 发布后先做哪些冒烟测试

至少做下面这些：

1. 普通用户登录
2. 管理员登录
3. 发布动态
4. 查看公开流
5. 进入动态审核页
6. 提交举报
7. 进入举报处理页
8. 查看通知中心

## 13. 微信小程序版本怎么发布

这是很多第一次做小程序的人最容易懵的地方。你可以把它理解成 4 步。

### 13.1 第一步：生成微信小程序代码

如果你使用命令行：

```bash
npm run build:mp-weixin
```

如果你使用 HBuilderX：

1. 点击“发行”
2. 选择“小程序-微信”
3. 选择“发行”

### 13.2 第二步：用微信开发者工具打开

详细步骤：

1. 打开微信开发者工具
2. 选择“导入项目”
3. 项目目录选择 HBuilderX 编译后的微信小程序目录
4. 选择对应的 `AppID`
5. 点击导入

### 13.3 第三步：先上传体验版

这是第一次上线前最建议做的。

详细步骤：

1. 在微信开发者工具里确认编译通过
2. 点击“上传”
3. 填写版本号
4. 填写项目备注
5. 上传成功后，登录微信公众平台
6. 进入“小程序后台”
7. 找到“版本管理”
8. 把这个版本设为体验版
9. 让自己和测试人员扫码体验

### 13.4 第四步：提交微信审核并发布

详细步骤：

1. 登录微信公众平台
2. 进入“小程序后台”
3. 打开“版本管理”
4. 找到刚上传的开发版本
5. 点击“提交审核”
6. 按要求填写：
   - 功能页面说明
   - 测试账号
   - 测试路径
   - 类目和服务说明
7. 审核通过后，点击“发布”

### 13.5 提交微信审核前一定要检查什么

1. 登录流程正常
2. 发布动态正常
3. 举报流程正常
4. 管理员功能不要暴露给普通用户
5. 隐私协议、用户协议准备好
6. 位置权限申请描述清晰
7. 不要出现测试文案、演示数据、空白页

## 14. 管理员日常运维怎么做

日常运维你可以理解成“发现异常后怎么查、怎么修、怎么确认修好了”。

## 15. 运维检查清单

建议你每天或每次版本发布后检查下面几项。

### 15.1 内容审核队列

检查内容：

1. 是否有长期未处理的待审核动态
2. 待审核数量是否突然暴增
3. 审核通过/驳回后，用户是否收到通知

### 15.2 举报处理队列

检查内容：

1. 是否有大量待处理举报
2. 是否出现重复举报集中在同一对象
3. 处置动作后，状态是否真的更新

### 15.3 通知中心

检查内容：

1. 审核通知是否正常写入
2. 举报处理通知是否正常写入
3. 未读数是否正常变化

### 15.4 云函数日志

重点看这些云函数：

1. `fit-discover-api`
2. `fit-report-api`
3. `fit-notification-api`
4. `fit-maintenance-api`

重点看：

1. 报错次数
2. 平均耗时
3. 是否出现大量超时
4. 是否出现鉴权失败

## 16. 聊天未读数不对时怎么修

项目里已经提供了专门的运维能力。

相关文档：

- `docs/chat-unread-rebuild.md`

### 16.1 什么情况需要执行

例如：

1. 用户反馈聊天红点数量不对
2. 会话列表未读数和实际消息数对不上
3. 历史数据迁移后未读数异常

### 16.2 修复前先确认

1. 执行账号必须是管理员
2. `fit-maintenance-api` 已部署
3. `fit-conversations` 和 `fit-messages` schema 已同步

### 16.3 建议的执行方式

先 dry run，再正式执行。

#### 第一步：先预演

```js
await uniCloud.callFunction({
  name: 'fit-maintenance-api',
  data: {
    action: 'rebuildConversationUnreadMap',
    params: {
      offset: 0,
      batchSize: 50,
      dryRun: true
    }
  }
})
```

看返回结果是否正常，重点看：

1. `processed`
2. `updated`
3. `unchanged`
4. `skipped`
5. `hasMore`

#### 第二步：正式执行

```js
await uniCloud.callFunction({
  name: 'fit-maintenance-api',
  data: {
    action: 'rebuildConversationUnreadMap',
    params: {
      offset: 0,
      batchSize: 100,
      dryRun: false
    }
  }
})
```

#### 第三步：循环执行直到完成

如果返回：

- `hasMore = true`

就继续用返回的 `nextOffset` 再跑下一批。

### 16.4 修复后怎么验证

1. 随机找几条历史会话
2. 检查未读数是否和实际一致
3. 新发消息后，未读数是否还能继续正常变化

## 17. 内容安全规则怎么运维

当前项目支持内容安全规则表：

- `fit-content-security-rules`

它可以作用于这些场景：

1. `all`
2. `feed`
3. `comment`
4. `invite`
5. `report`

命中后的处理模式：

1. `review`
2. `block`

### 17.1 你可以怎么理解

如果某个词命中规则：

1. `review`：进入人工审核
2. `block`：直接拦截

### 17.2 目前怎么改

当前没有前台管理页，通常是在数据库里直接维护。

操作步骤：

1. 打开数据库 `fit-content-security-rules`
2. 修改或新增规则
3. 设置：
   - `scene`
   - `word`
   - `match_mode`
   - `mode`
   - `priority`
   - `enabled`
4. 保存
5. 过 1 分钟左右再验证

说明：

- 云函数里对规则有缓存，缓存时间约 60 秒
- 所以你改完后，不一定立刻生效

## 18. 常见问题排查

## 18.1 看不到管理员面板

按顺序排查：

1. 当前账号是否真的有 `admin` 角色
2. 修改角色后是否重新登录
3. 当前连接的是不是正确服务空间
4. `uni-id-users` 是否同步到了正确环境

## 18.2 管理员能进页面，但操作报“仅管理员可执行该操作”

说明前端看起来像管理员，但云函数鉴权没通过。

排查顺序：

1. 当前登录 token 是否还是旧的
2. 用户 `role` 字段是否真的包含 `admin`
3. 云函数是否部署到了正确服务空间
4. 是否前端连的是 A 环境，云函数发到了 B 环境

## 18.3 用户发布动态后自己看不到

通常是下面几种原因：

1. 动态进入待审核了
2. 动态被驳回了
3. 动态被举报后下架了

排查方法：

1. 去 `fit-feeds` 看 `status`
2. 看 `moderation_status`
3. 看 `review_remark`

## 18.4 举报处理后，用户没收到通知

排查方法：

1. 查看 `fit-notifications` 是否写入成功
2. 检查 `toUserId` 是否正确
3. 查看 `fit-notification-api` 是否正常
4. 前端通知中心是否能正常拉取

## 18.5 发布后线上和本地表现不一致

优先排查：

1. 服务空间是否一致
2. `manifest.json` 的 `mp-weixin.appid` 是否一致
3. schema 是否同步
4. 云函数是否都重新部署
5. 微信体验版是不是最新上传的版本

## 19. 适合你的实际操作顺序

如果你现在就要接手这个项目，我建议你严格照下面顺序来。

### 第一天先做

3. 确认云函数已部署

1. 每天看待审核动态

## 20. 你现在最该记住的 3 件事
